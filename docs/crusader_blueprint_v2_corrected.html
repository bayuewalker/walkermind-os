Execution Matrix.

CrusaderBot is a multi-user, non-custodial prediction market trading system built around a strict separation of concerns: Telegram as the user control surface, FastAPI as the control-plane backend, Falcon MCP as the read-side intelligence layer, and Polymarket CLOB as the write-side execution venue. Users link wallets, authorize sessions, and control automation through Telegram; Crusader does not take custody of user funds. The system is designed to run continuously, evaluate opportunities automatically, enforce risk synchronously, execute only through approved paths, and keep every user scope isolated.

Principle 01
Risk-First

A profitable-looking signal is still rejected if it breaches drawdown, exposure, liquidity, or execution-safety constraints. Risk is enforced in code, not implied by documentation.

Principle 02
Control, Not Manual Entry

Users do not place discretionary one-off trades via Telegram. Users control automation, safety, mode, and monitoring through Telegram while the backend remains the decision authority.

Principle 03
Read / Write Separation

Falcon enriches intelligence and discovery. CLOB executes. They do not overlap. This boundary is one of the core safety guarantees of the system.

Read / Write Boundary

Falcon is strictly read-only intelligence: markets, candles, trades, wallet analytics, social pulse, and anomaly signals. It enriches strategy evaluation and portfolio visibility. It does not place, cancel, or manage orders.

Execution Authority

All trade writes route through the backend into Polymarket CLOB after signal approval and risk validation. Telegram does not act as a manual order terminal.

Signal Engine

FeatureBuilder, ProbabilityModel, EdgeEngine, EVEngine, regime filters, and strategy selectors turn raw market data into ranked opportunities.

Risk Engine

SizingEngine, fractional Kelly, exposure controls, drawdown guards, user-tier limits, and trade approval gates enforce capital preservation.

Execution Engine

Order planning, routing, fill handling, slippage checks, settlement tracking, and idempotent guards keep execution deterministic and safe.

Security Protocol: intelligence layers are read-only, execution lives behind backend risk approval, and user controls are separated from trade authority.

Crusader prioritizes capital preservation over alpha. The Risk Engine is the synchronous approval gate between signal generation and execution.

Enforced Logic Rules

·
Positive EV Gate: orders rejected if EV is not positive after slippage and fee modeling.
·
Minimum Edge: mispricing must pass configured edge threshold before a trade becomes eligible.
·
Liquidity Floor: target market must meet minimum executable depth before entry approval.
·
Idempotency: duplicate intent hashing prevents repeated order submission within the same event window.
·
MDD Auto-Pause: live auto-trading is disabled automatically if session loss exceeds threshold.
·
Cross-User Isolation: all user-facing activity and execution state must be scoped by user, account, or wallet boundary.

Safety & Failure
  • Stale market data → safe mode fallback
  • Falcon unavailable → retry and cached degradation
  • Bankroll desync → HALT and resync
  • Position mismatch → reconcile before RESUME
